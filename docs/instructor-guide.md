# 강사 가이드

이 문서는 강사가 재직자 AI 서비스 개발 과정에서 `codex-ai-service-kit`을 운영하기 위한 가이드입니다.

이 Kit의 핵심 목적은 수강생이 Codex를 단순 코드 생성 도구로 사용하는 것이 아니라, 요구사항 정의부터 구현, 오류 분석, 테스트, 문서화까지 일관된 개발 흐름을 경험하도록 돕는 것입니다.

## 1. Kit 운영 목적

`codex-ai-service-kit`은 모든 수강생이 공통된 개발 절차를 따르도록 지원합니다.

| 운영 목적 | 설명 |
| --- | --- |
| 개발 흐름 표준화 | 요구사항 정의, 기능 분해, MVP 설계, 구현, 테스트, 문서화 순서를 따르게 합니다. |
| Codex 사용 품질 향상 | 막연한 요청 대신 단계별 프롬프트를 사용하도록 지도합니다. |
| MVP 중심 개발 | 수업 시간 안에 실행 가능한 결과물을 만들도록 범위를 줄입니다. |
| 산출물 품질 향상 | README, 보고서, 테스트 결과 등 제출 자료를 체계화합니다. |
| 보안 습관 형성 | API Key, token, `.env` 등 민감정보를 GitHub에 올리지 않도록 지도합니다. |

## 2. 수업 전 배포 준비

수업 전에 아래 항목을 준비합니다.

| 준비 항목 | 확인 내용 |
| --- | --- |
| GitHub 저장소 | `codex-ai-service-kit` 저장소가 접근 가능한지 확인합니다. |
| 설치 스크립트 | Windows용 `scripts/install.ps1`, macOS/Linux용 `scripts/install.sh`가 포함되어 있는지 확인합니다. |
| 학생 가이드 | `docs/student-guide.md`를 수강생에게 안내합니다. |
| 예시 프로젝트 | 간단한 AI 서비스 예시를 하나 준비하면 좋습니다. |
| API Key 안내 | OpenAI API 또는 Gemini API Key 발급 방법과 보안 주의사항을 안내합니다. |
| 실습 환경 | Python, Git, Codex 실행 환경을 확인합니다. |

배포 방식 예시:

```text
1. GitHub 저장소 링크 공유
2. student-guide.md 읽기 안내
3. 설치 명령 실행
4. 설치 확인
5. 첫 요구사항 정의 실습 진행
```

## 3. 수강생 설치 확인 방법

Windows 기준 설치 명령:

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\install.ps1
```

설치 확인 명령:

```powershell
dir $HOME\.codex
dir $HOME\.codex\skills
```

확인해야 할 항목:

| 항목 | 기대 결과 |
| --- | --- |
| `~/.codex/AGENTS.md` | 파일이 존재해야 합니다. |
| `~/.codex/agents/` | `routing.md`가 있어야 합니다. |
| `~/.codex/skills/` | 10개 Skill 폴더가 있어야 합니다. |
| `~/.codex/config/` | `lean-skills.txt`, `codex.config.sample.toml`이 있어야 합니다. |

설치가 실패한 경우에는 PowerShell 실행 위치, 권한, 저장소 압축 해제 여부를 먼저 확인합니다.

## 4. 첫 실습 진행 순서

첫 실습은 구현보다 요구사항 정의와 MVP 범위 조정에 집중하는 것이 좋습니다.

권장 진행 순서:

| 순서 | 활동 | 목표 |
| --- | --- | --- |
| 1 | Kit 설치 확인 | 모든 수강생의 Codex 설정 준비 |
| 2 | 프로젝트 아이디어 작성 | 만들 서비스의 주제 정하기 |
| 3 | 요구사항 정의 프롬프트 사용 | 아이디어를 요구사항 정의서로 변환 |
| 4 | 기능 분해 프롬프트 사용 | 구현 가능한 기능 단위로 나누기 |
| 5 | MVP 설계 프롬프트 사용 | 수업 시간 안에 만들 범위 정하기 |
| 6 | 프로젝트 구조 생성 | FastAPI 중심 기본 구조 만들기 |
| 7 | 첫 기능 구현 | 가장 작은 핵심 기능 하나 구현 |

첫 실습에서 피해야 할 것:

- 처음부터 전체 서비스를 만들게 하기
- 로그인, 결제, 관리자 기능을 바로 넣기
- README와 보안 안내를 마지막까지 미루기
- 오류 로그 없이 디버깅하게 하기

## 5. 공통 프롬프트 사용 지도법

수강생에게 `prompts/` 폴더의 파일을 복사해서 사용하도록 안내합니다.

| 단계 | 프롬프트 파일 |
| --- | --- |
| 요구사항 정의 | `prompts/01-requirements.md` |
| 기능 분해 | `prompts/02-function-breakdown.md` |
| MVP 설계 | `prompts/03-mvp-planning.md` |
| 프로젝트 구조 | `prompts/04-project-structure.md` |
| 기능 구현 | `prompts/05-feature-build.md` |
| 오류 분석 | `prompts/06-debugging.md` |
| 리팩토링 | `prompts/07-refactoring.md` |
| README / 보고서 | `prompts/08-readme-report.md` |
| 보안 점검 | `prompts/09-security-check.md` |

지도 포인트:

- 빈칸을 최대한 구체적으로 채우게 합니다.
- "전체 만들어줘"보다 "이 기능 하나를 구현해줘"라고 요청하게 합니다.
- Codex 답변을 그대로 믿지 말고 실행과 테스트로 확인하게 합니다.
- 산출물은 `templates/` 양식에 맞춰 정리하게 합니다.

## 6. 프로젝트별 적용 방식

이 Kit는 특정 프로젝트 템플릿이 아니라 공통 개발 절차 Kit입니다.

프로젝트별 적용 기준:

| 프로젝트 유형 | 적용 방식 |
| --- | --- |
| 요약 서비스 | 입력 데이터, 요약 기준, 출력 형식을 먼저 정의합니다. |
| 추천 서비스 | 추천 대상, 사용자 조건, 추천 결과 형식을 명확히 합니다. |
| 챗봇 서비스 | 질문 유형, 답변 기준, 검색 필요 여부를 구분합니다. |
| 문서 검색 서비스 | 문서 입력, 임베딩, 벡터 검색, 답변 생성을 단계화합니다. |
| 분류 서비스 | 분류 기준, 라벨, 예시 데이터를 먼저 정합니다. |

강사는 모든 프로젝트에 같은 질문을 던지면 좋습니다.

```text
이 서비스의 핵심 입력은 무엇인가?
AI는 무엇을 처리하는가?
사용자는 어떤 결과를 확인하는가?
MVP에서 반드시 실행되어야 하는 흐름은 무엇인가?
```

## 7. 디버깅 지도 방식

디버깅은 정답을 바로 알려주기보다 로그 읽는 방법을 가르치는 것이 중요합니다.

지도 순서:

| 순서 | 확인 항목 |
| --- | --- |
| 1 | 어떤 명령을 실행했는가? |
| 2 | 전체 오류 로그는 무엇인가? |
| 3 | 마지막 에러 메시지는 무엇인가? |
| 4 | 파일 경로와 줄 번호가 있는가? |
| 5 | 관련 코드를 확인했는가? |
| 6 | 최소 수정으로 해결 가능한가? |
| 7 | 재실행 결과가 어떻게 바뀌었는가? |

수강생에게 권장할 요청:

```text
아래 오류를 로그와 코드 기준으로 분석해줘.
추측하지 말고 오류 메시지, 발생 위치, 관련 파일을 근거로 원인을 설명해줘.
```

디버깅 지도 시 주의:

- 로그 없이 원인을 단정하지 않게 합니다.
- 오류가 나왔다고 전체 구조를 바꾸지 않게 합니다.
- 수정 후 재실행 명령과 확인 방법을 반드시 남기게 합니다.

## 8. 코드 리뷰 기준

코드 리뷰는 완벽한 설계보다 수업 목표와 MVP 완성도를 기준으로 봅니다.

리뷰 기준:

| 기준 | 확인 질문 |
| --- | --- |
| 기능 동작 | MVP 핵심 흐름이 실행되는가? |
| 구조 | router, service, schema 역할이 과도하게 섞이지 않았는가? |
| 가독성 | 수강생이 설명할 수 있는 코드인가? |
| 오류 처리 | 기본 오류 상황을 처리하는가? |
| 보안 | API Key나 `.env` 값이 코드에 직접 들어가 있지 않은가? |
| 테스트 | 실행 또는 수동 확인 절차가 있는가? |
| 문서화 | README에 실행 방법과 주요 기능이 정리되어 있는가? |

리뷰 코멘트는 구체적으로 작성합니다.

좋은 코멘트:

```text
`summarize.py` router 안에 AI 호출 로직이 직접 들어가 있어요.
다음 단계에서는 이 부분을 `services/summarize_service.py`로 분리하면 역할이 더 명확해집니다.
```

피해야 할 코멘트:

```text
구조가 별로예요.
```

제출 산출물은 코드뿐 아니라 과정 문서도 함께 평가합니다.

권장 평가 항목:

| 평가 항목 | 배점 예시 | 확인 기준 |
| --- | --- | --- |
| 요구사항 정의 | 15 | 문제, 사용자, 입력, 출력이 명확한가? |
| 기능 분해 | 10 | 구현 가능한 단위로 나누었는가? |
| MVP 설계 | 15 | 수업 시간 안에 실행 가능한 범위인가? |
| 구현 결과 | 25 | 핵심 기능이 실제로 동작하는가? |
| 오류 분석과 테스트 | 10 | 오류 해결 과정과 테스트 결과가 기록되어 있는가? |
| README | 10 | 설치, 실행, API 사용 방법이 명확한가? |
| 최종 보고서 | 10 | 과정, 결과, 한계, 개선 사항이 정리되어 있는가? |
| 보안 점검 | 5 | 민감정보가 포함되지 않았는가? |

필수 제출물 예시:

```text
README.md
docs/requirements.md
docs/function-breakdown.md
docs/mvp-plan.md
docs/test-scenario.md
docs/final-report.md
```

## 10. 보안 지도 사항

AI 서비스 수업에서는 API Key와 사용자 데이터 보안 지도가 중요합니다.

반드시 강조할 항목:

| 항목 | 지도 내용 |
| --- | --- |
| `.env` | GitHub에 올리지 않습니다. |
| `.env.example` | 가짜 예시 값만 작성합니다. |
| API Key | 코드, README, 보고서에 직접 쓰지 않습니다. |
| token/password | 채팅, 이슈, 커밋에 남기지 않습니다. |
| `credentials.json` | GitHub에 올리지 않습니다. |
| `auth.json` | GitHub에 올리지 않습니다. |
| 고객 데이터 | 가명 처리하거나 샘플 데이터로 대체합니다. |

GitHub 업로드 전 공통 요청:

```text
GitHub에 push하기 전에 민감정보와 보안 위험을 점검해줘.
API Key, token, password, credentials.json, auth.json, .env 파일이 포함되어 있는지 확인해줘.
민감정보가 발견되면 실제 값은 출력하지 말고 파일 위치와 조치 방법만 알려줘.
```

이미 API Key가 올라간 경우:

1. 해당 키를 즉시 폐기합니다.
2. 새 키를 발급합니다.
3. 저장소에서 민감정보를 제거합니다.
4. 같은 실수가 반복되지 않도록 `.gitignore`를 확인합니다.

## 11. 수업 중 자주 발생하는 문제와 대응법

| 문제 | 원인 | 대응법 |
| --- | --- | --- |
| 설치 스크립트가 실행되지 않음 | PowerShell 실행 정책 또는 경로 문제 | `powershell -ExecutionPolicy Bypass -File .\scripts\install.ps1`로 실행하게 합니다. |
| `~/.codex`에 파일이 없음 | 설치 위치가 잘못되었거나 스크립트 실행 실패 | `dir $HOME\.codex`로 확인하고 다시 설치합니다. |
| 수강생이 전체 서비스를 한 번에 요청함 | 개발 범위를 줄이는 경험 부족 | 요구사항 정의와 MVP 설계부터 다시 진행시킵니다. |
| 오류 로그 없이 질문함 | 에러 메시지의 중요성을 모름 | 실행 명령, 전체 로그, 관련 파일을 함께 보내게 합니다. |
| API Key가 코드에 들어감 | 환경 변수 사용 경험 부족 | `.env`와 `.env.example` 차이를 다시 설명합니다. |
| README가 부실함 | 문서화를 마지막 작업으로만 생각함 | 중간 점검 때 README 초안을 제출하게 합니다. |
| 기능이 너무 많아 미완성됨 | MVP 범위 설정 실패 | 핵심 입력, AI 처리, 출력 결과만 남기게 합니다. |
| Codex 답변을 그대로 복사함 | 검증 습관 부족 | 반드시 실행과 테스트 결과를 기록하게 합니다. |
| 팀원별 코드가 충돌함 | 역할 분담과 파일 소유가 불명확함 | 기능 단위로 담당자를 나누고 수정 파일을 공유하게 합니다. |

강사의 핵심 역할은 Codex가 만든 결과를 대신 검증해주는 것이 아니라, 수강생이 요구사항, 범위, 실행, 테스트, 문서화의 흐름을 스스로 익히도록 돕는 것입니다.
