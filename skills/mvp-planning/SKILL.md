# MVP Planning

## 1. 목적

이 Skill의 목적은 전체 프로젝트 중 수업 시간 안에 먼저 구현할 최소 기능 범위를 정하는 것입니다.

MVP는 단순한 기획 목록이 아니라, 사용자가 실제로 실행하고 결과를 확인할 수 있는 최소 버전이어야 합니다. Codex는 많은 기능을 한 번에 구현하려 하지 않고, 핵심 문제를 해결하는 실행 가능한 결과물을 우선 만들도록 범위를 조정해야 합니다.

이 Skill의 결과물은 프로젝트 구조 생성, 기능 구현, 테스트, README 작성의 기준으로 사용됩니다.

## 2. 사용 상황

다음과 같은 상황에서 이 Skill을 사용합니다.

- 요구사항 정의와 기능 분해가 끝난 후 첫 구현 범위를 정해야 할 때
- 수업 시간 또는 제출 기한 안에 구현 가능한 범위를 정해야 할 때
- 기능 목록이 너무 많아 무엇부터 만들어야 할지 모를 때
- 팀 프로젝트에서 우선 구현할 기능을 합의해야 할 때
- 사용자가 전체 서비스를 한 번에 만들려고 할 때
- 실행 가능한 첫 번째 버전을 빠르게 만들고 싶을 때

예시 요청:

```text
기능이 너무 많은데 수업 시간 안에 만들 MVP 범위를 정해줘.
```

## 3. MVP를 정하는 기준

MVP는 Minimum Viable Product의 약자로, 서비스의 핵심 가치를 검증할 수 있는 최소 실행 버전입니다.

이 Kit에서 MVP는 반드시 다음 조건을 만족해야 합니다.

- 사용자가 직접 실행할 수 있습니다.
- 입력을 넣으면 처리 결과를 확인할 수 있습니다.
- 서비스의 핵심 문제를 최소한으로 해결합니다.
- 수업 시간 안에 구현 가능한 범위입니다.
- README에 실행 방법을 설명할 수 있습니다.
- 간단한 테스트 또는 수동 확인이 가능합니다.

MVP가 아닌 것:

- 기능 아이디어 목록만 있는 상태
- 화면만 있고 실제 처리가 없는 상태
- 백엔드만 있고 사용자가 결과를 확인할 수 없는 상태
- AI 호출 코드만 있고 서비스 흐름이 없는 상태
- 실행 방법을 설명할 수 없는 상태

## 4. MVP에 포함할 기능 기준

다음 기준에 해당하는 기능은 MVP에 포함할 수 있습니다.

- 없으면 서비스의 핵심 가치가 사라지는 기능
- 사용자의 입력을 받을 수 있게 하는 기능
- 입력 데이터를 처리하는 핵심 AI 기능
- 처리 결과를 사용자에게 보여주는 기능
- 기본적인 오류 메시지를 제공하는 기능
- 실행과 확인에 필요한 최소 API 또는 화면 기능
- README에 사용 예시로 보여줄 수 있는 기능

MVP 기능은 다음 질문에 "예"라고 답할 수 있어야 합니다.

- 이 기능이 없으면 서비스가 동작하지 않는가?
- 사용자가 결과를 확인하는 데 필요한가?
- 수업 시간 안에 구현 가능한가?
- 테스트하거나 수동으로 확인할 수 있는가?
- 프로젝트의 핵심 문제 해결과 직접 관련이 있는가?

예시:

| 기능 | MVP 포함 여부 | 이유 |
| --- | --- | --- |
| 회의록 텍스트 입력 | 포함 | 사용자의 입력이 있어야 서비스가 시작됩니다. |
| AI 요약 생성 | 포함 | 서비스의 핵심 기능입니다. |
| 요약 결과 표시 | 포함 | 사용자가 결과를 확인해야 합니다. |
| 빈 입력값 오류 메시지 | 포함 | 최소한의 사용성에 필요합니다. |

## 5. MVP에서 제외할 기능 기준

다음 기준에 해당하는 기능은 MVP에서 제외하고 후순위로 둡니다.

- 없어도 핵심 기능이 실행되는 기능
- 구현 시간이 많이 걸리는 기능
- 화면 편의성이나 디자인 개선 중심 기능
- 운영, 배포, 관리자 기능
- 로그인, 결제, 권한 관리처럼 초기 검증에 필수적이지 않은 기능
- 고급 통계, 대시보드, 알림, 즐겨찾기 등 부가 기능
- 실패해도 대체 방식으로 시연 가능한 기능

제외할 때는 단순히 버리지 않고 "후순위 기능"으로 기록합니다.

예시:

| 기능 | MVP 제외 여부 | 이유 |
| --- | --- | --- |
| 사용자 로그인 | 제외 | 첫 버전에서는 개인화 없이도 핵심 기능을 검증할 수 있습니다. |
| PDF 다운로드 | 제외 | 화면에서 결과 확인이 가능하므로 필수는 아닙니다. |
| 관리자 대시보드 | 제외 | 수업 MVP의 핵심 사용자 흐름과 거리가 있습니다. |
| 배포 자동화 | 제외 | 로컬 실행으로도 MVP 검증이 가능합니다. |

## 6. 구현 순서 정리 방식

Codex는 MVP 기능을 다음 순서로 정리합니다.

1. 가장 단순한 실행 흐름을 먼저 만듭니다.
2. 사용자 입력 기능을 먼저 구현합니다.
3. 핵심 처리 기능을 연결합니다.
4. 결과 출력 기능을 구현합니다.
5. 최소 오류 처리를 추가합니다.
6. 테스트 또는 수동 확인 절차를 작성합니다.
7. README에 실행 방법을 정리합니다.

구현 순서는 기능 간 의존성을 기준으로 정해야 합니다.

예시 순서:

```text
입력 화면
→ 요청 API
→ AI 처리 함수
→ 결과 반환
→ 결과 화면 표시
→ 오류 처리
→ 테스트
→ README 정리
```

각 구현 단계는 가능하면 1개의 작업 요청으로 처리할 수 있을 정도로 작게 나눕니다.

## 7. 완료 기준 정의 방식

MVP 완료 기준은 "구현했다"가 아니라 "실행해서 결과를 확인할 수 있다"를 기준으로 정의합니다.

완료 기준에는 다음 내용을 포함합니다.

- 실행 명령이 있습니다.
- 사용자가 입력할 예시가 있습니다.
- 기대 출력 결과가 있습니다.
- 핵심 기능이 정상 동작합니다.
- 최소 오류 상황을 확인할 수 있습니다.
- README에 실행 방법이 정리되어 있습니다.

좋은 완료 기준 예시:

```text
사용자가 회의록 텍스트를 입력하면, 서비스가 핵심 요약과 할 일 목록을 화면에 표시한다.
빈 텍스트를 입력하면 오류 메시지를 표시한다.
README의 실행 명령을 따라 로컬에서 서비스를 실행할 수 있다.
```

나쁜 완료 기준 예시:

```text
AI 회의록 요약 기능 구현 완료
```

문제점:

- 실행 가능 여부가 없습니다.
- 입력과 출력이 명확하지 않습니다.
- 오류 상황과 확인 방법이 없습니다.

## 9. 예시

아래는 "AI 회의록 요약 서비스"의 MVP 계획 예시입니다.

```markdown
# MVP 계획서

## 1. MVP 목표

- 프로젝트명: AI 회의록 요약 서비스
- MVP 한 줄 설명: 회의록 텍스트를 입력하면 핵심 요약과 할 일 목록을 보여주는 서비스
- MVP에서 검증할 핵심 가치: 긴 회의록을 빠르게 이해하고 후속 조치를 확인할 수 있는지 검증합니다.

## 2. MVP 사용자 흐름

1. 사용자가 회의록 텍스트를 입력합니다.
2. 서비스가 입력값이 비어 있는지 확인합니다.
3. AI가 핵심 안건, 결정 사항, 할 일 목록을 추출합니다.
4. 서비스가 결과를 화면에 보기 좋게 정리합니다.
5. 사용자가 요약 결과를 확인합니다.

## 3. MVP에 포함할 기능

| 번호 | 기능 | 설명 | 포함 이유 |
| --- | --- | --- | --- |
| 1 | 회의록 텍스트 입력 | 사용자가 회의록 내용을 입력합니다. | 입력이 있어야 서비스가 시작됩니다. |
| 2 | AI 요약 생성 | 회의록에서 핵심 안건과 결정 사항을 요약합니다. | 서비스의 핵심 기능입니다. |
| 3 | 할 일 목록 추출 | 회의록에서 후속 작업을 추출합니다. | 사용자가 바로 행동할 수 있는 결과를 제공합니다. |
| 4 | 결과 화면 표시 | 요약과 할 일 목록을 화면에 표시합니다. | 사용자가 결과를 확인해야 합니다. |
| 5 | 기본 오류 처리 | 빈 입력값일 때 안내 메시지를 보여줍니다. | 최소한의 사용성에 필요합니다. |

## 4. MVP에서 제외할 기능

| 번호 | 기능 | 제외 이유 | 후순위 처리 방향 |
| --- | --- | --- | --- |
| 1 | 사용자 로그인 | 첫 버전에서는 개인화가 없어도 핵심 기능 검증이 가능합니다. | 결과 저장 기능 이후 검토합니다. |
| 2 | PDF 다운로드 | 화면에서 결과 확인이 가능하므로 필수는 아닙니다. | 문서 공유 기능이 필요할 때 추가합니다. |
| 3 | 회의록 파일 업로드 | 텍스트 입력만으로 MVP 검증이 가능합니다. | 텍스트 입력이 안정화된 후 추가합니다. |

## 5. 구현 순서

| 순서 | 작업 | 결과물 | 확인 방법 |
| --- | --- | --- | --- |
| 1 | 텍스트 입력 화면 만들기 | 입력 폼 | 브라우저에서 텍스트를 입력할 수 있습니다. |
| 2 | 요약 요청 API 만들기 | `/summarize` API | 샘플 텍스트 요청에 응답합니다. |
| 3 | AI 요약 함수 연결 | 요약 처리 함수 | 핵심 요약과 할 일 목록이 생성됩니다. |
| 4 | 결과 화면 표시 | 결과 영역 | 입력 후 결과가 화면에 표시됩니다. |
| 5 | 오류 처리 추가 | 오류 메시지 | 빈 입력값에서 안내 메시지가 표시됩니다. |
| 6 | README 작성 | README.md | 실행 방법과 사용 예시가 문서화됩니다. |

## 6. 완료 기준

- 사용자가 회의록 텍스트를 입력할 수 있습니다.
- AI가 핵심 요약과 할 일 목록을 생성합니다.
- 결과가 화면에 표시됩니다.
- 빈 입력값을 넣으면 오류 메시지가 표시됩니다.
- README를 보고 로컬에서 실행할 수 있습니다.

## 7. 테스트 또는 수동 확인 방법

- 실행 명령: 프로젝트 README의 실행 명령을 따릅니다.
- 입력 예시: 회의록 샘플 텍스트 5문장 이상
- 기대 결과: 핵심 요약 3개와 할 일 목록이 표시됩니다.
- 오류 상황 확인: 빈 텍스트 입력 시 안내 메시지가 표시됩니다.
```
