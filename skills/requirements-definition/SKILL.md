# Requirements Definition

## 1. 목적

이 Skill의 목적은 사용자의 아이디어를 AI 서비스 개발용 요구사항 정의서로 바꾸는 것입니다.

Codex는 사용자가 말한 서비스 아이디어를 바탕으로 프로젝트 목표, 해결 문제, 대상 사용자, 입력 데이터, 핵심 기능, 처리 흐름, 출력 결과, MVP 범위, 추가 기능을 구조화해야 합니다.

이 Skill은 바로 구현을 시작하기 전에 사용합니다. 요구사항이 명확해야 이후 기능 분해, MVP 설계, 프로젝트 구조 생성, 구현, 테스트, README 작성이 일관되게 진행될 수 있습니다.

## 2. 사용 상황

다음과 같은 상황에서 이 Skill을 사용합니다.

- 사용자가 AI 서비스 아이디어만 가지고 있는 경우
- 프로젝트 주제는 정했지만 구체적인 기능이 정리되지 않은 경우
- 구현 전에 요구사항 정의서가 필요한 경우
- 팀 프로젝트에서 공통 이해가 필요한 경우
- 수업 제출용 기획서 또는 보고서의 기초 자료가 필요한 경우
- 사용자가 "무엇부터 해야 할지 모르겠다"고 말하는 경우
- 사용자가 바로 전체 구현을 요청했지만 범위가 너무 큰 경우

예시 상황:

```text
AI로 고객 리뷰를 분석하는 서비스를 만들고 싶어.
```

이 경우 Codex는 바로 코드를 만들지 않고, 먼저 요구사항 정의서를 작성해야 합니다.

## 3. 입력 정보

사용자가 제공할 수 있는 입력 정보는 다음과 같습니다.

- 서비스 아이디어
- 해결하고 싶은 문제
- 예상 사용자
- 사용자가 입력할 데이터
- 원하는 결과물
- 사용할 AI 기능
- 사용하고 싶은 기술 스택
- 프로젝트 제출 조건
- 개발 기간
- 팀원 수

모든 정보가 처음부터 제공되지 않아도 됩니다. 정보가 부족하면 Codex는 합리적인 가정을 명시하고, 필요한 경우 추가 질문을 제시합니다.

필수로 확인해야 하는 정보:

- 어떤 문제를 해결하려는가?
- 누구를 위한 서비스인가?
- 사용자는 무엇을 입력하는가?
- 서비스는 어떤 결과를 제공하는가?
- 첫 번째 버전에서 반드시 필요한 기능은 무엇인가?

## 4. 작업 절차

Codex는 다음 순서로 작업합니다.

1. 사용자의 아이디어를 한 문장으로 요약합니다.
2. 서비스가 해결하려는 문제를 정리합니다.
3. 대상 사용자를 구체화합니다.
4. 입력 데이터와 출력 결과를 구분합니다.
5. 핵심 기능을 3개에서 7개 정도로 정리합니다.
6. 사용자의 입력부터 결과 출력까지의 처리 흐름을 단계별로 작성합니다.
7. MVP에 포함할 기능과 제외할 기능을 나눕니다.
8. 추가 기능 또는 향후 개선 기능을 정리합니다.
9. 모호한 부분은 "가정" 또는 "확인 필요 사항"으로 표시합니다.
10. 최종 산출물을 요구사항 정의서 형식으로 제공합니다.

작업 중 지켜야 할 원칙:

- 바로 구현하지 않습니다.
- 기능을 너무 많이 확정하지 않습니다.
- 수강생이 이해할 수 있는 쉬운 표현을 사용합니다.
- 기술 용어가 필요하면 짧게 설명합니다.
- MVP 범위를 작고 현실적으로 잡습니다.

## 5. 산출물 형식

Codex는 요구사항 정의 결과를 Markdown 문서 형식으로 작성합니다.

산출물에는 반드시 다음 항목을 포함해야 합니다.

- 프로젝트 개요
- 해결하고자 하는 문제
- 대상 사용자
- 입력 데이터
- 핵심 기능
- 처리 흐름
- 출력 결과
- MVP 범위
- 추가 기능

필요한 경우 다음 항목을 추가할 수 있습니다.

- 가정 사항
- 확인 필요 사항
- 제외 범위
- 평가 기준
- 예상 기술 스택

## 7. 좋은 요청 예시

좋은 요청은 서비스 아이디어, 사용자, 입력 데이터, 원하는 결과가 어느 정도 포함되어 있습니다.

예시:

```text
AI 회의록 요약 서비스를 만들고 싶어.
사용자는 회의 녹취록을 가진 직장인이야.
텍스트 파일을 업로드하면 핵심 안건, 결정 사항, 할 일 목록을 요약해주는 서비스로 만들고 싶어.
먼저 MVP 요구사항 정의서를 작성해줘.
```

이 요청이 좋은 이유:

- 서비스 주제가 명확합니다.
- 대상 사용자가 있습니다.
- 입력 데이터가 있습니다.
- 원하는 출력 결과가 있습니다.
- MVP 기준으로 정리해 달라고 요청했습니다.

## 8. 나쁜 요청 예시

나쁜 요청은 범위가 너무 크거나, 목표와 사용자가 불명확합니다.

예시:

```text
AI 서비스 하나 만들어줘.
```

문제점:

- 어떤 문제를 해결하는지 알 수 없습니다.
- 대상 사용자가 없습니다.
- 입력 데이터와 출력 결과가 없습니다.
- MVP 범위를 정할 수 없습니다.

이 경우 Codex는 바로 구현하지 말고 다음과 같이 응답해야 합니다.

```text
먼저 요구사항을 정리해야 합니다.
어떤 사용자를 위한 AI 서비스인지, 사용자가 무엇을 입력하고 어떤 결과를 받고 싶은지부터 정리해보겠습니다.
```

## 9. Codex가 주의할 점

- 사용자의 아이디어가 모호하면 임의로 확정하지 말고 가정을 명시합니다.
- 전체 시스템을 한 번에 설계하지 않습니다.
- 처음부터 로그인, 결제, 관리자 페이지, 배포 자동화 같은 부가 기능을 MVP에 넣지 않습니다.
- 수강생이 이해하기 어려운 전문 용어를 남용하지 않습니다.
- 구현 코드 작성으로 넘어가기 전에 요구사항 정의가 충분한지 확인합니다.
- 입력 데이터, 처리 흐름, 출력 결과를 반드시 구분합니다.
- 핵심 기능과 추가 기능을 섞지 않습니다.
- MVP 범위는 수업 기간 안에 구현 가능한 수준으로 제한합니다.
- 보안상 민감한 정보가 요구사항 문서에 포함되지 않도록 주의합니다.
- 최종 요구사항 정의서는 이후 기능 분해와 MVP 설계에 바로 사용할 수 있는 형태로 작성합니다.
