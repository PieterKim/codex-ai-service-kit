# README Report Writer

## 1. 목적

이 Skill의 목적은 프로젝트 제출용 README와 최종 보고서를 작성하는 것입니다.

Codex는 구현된 AI 서비스 프로젝트를 다른 사람이 이해하고 실행할 수 있도록 README를 작성하고, 수업 제출에 필요한 개발 과정, 기술 선택, 구현 결과, 테스트 결과, 한계점, 향후 개선 사항을 최종 보고서로 정리해야 합니다.

README는 "실행과 사용을 위한 문서"이고, 최종 보고서는 "프로젝트 과정과 결과를 설명하는 문서"입니다.

## 2. 사용 상황

다음과 같은 상황에서 이 Skill을 사용합니다.

- 프로젝트 제출 전 README를 정리해야 할 때
- 프로젝트의 실행 방법과 API 사용 방법을 문서화해야 할 때
- 최종 보고서 초안이 필요할 때
- 기술 스택, 폴더 구조, 테스트 결과를 정리해야 할 때
- 팀 프로젝트에서 역할과 구현 내용을 설명해야 할 때
- GitHub 저장소를 공개하기 전에 기본 문서를 정리해야 할 때
- 수강생이 프로젝트를 어떻게 설명해야 할지 모를 때

예시 요청:

```text
이 프로젝트를 제출할 수 있도록 README와 최종 보고서 초안을 작성해줘.
```

## 3. README 작성 기준

README는 프로젝트를 처음 보는 사람이 서비스를 이해하고 실행할 수 있게 작성합니다.

README에 포함할 기본 항목:

- 프로젝트명
- 프로젝트 소개
- 주요 기능
- 기술 스택
- 프로젝트 구조
- 설치 방법
- 환경 변수 설정 방법
- 실행 방법
- API 사용 방법
- 테스트 방법
- 사용 예시
- 주의사항
- 향후 개선 사항

README 작성 원칙:

- 실행 가능한 명령어를 포함합니다.
- 실제 API Key, 토큰, 비밀번호를 작성하지 않습니다.
- `.env.example` 기준으로 환경 변수를 안내합니다.
- 너무 긴 개발 일지는 README가 아니라 보고서에 작성합니다.
- 사용자가 따라 할 순서대로 작성합니다.
- 스크린샷이나 예시 응답이 있으면 이해하기 쉽습니다.

## 4. 최종 보고서 작성 기준

최종 보고서는 프로젝트의 목적, 개발 과정, 구현 내용, 결과, 배운 점을 설명하는 문서입니다.

최종 보고서에 포함할 기본 항목:

- 프로젝트 개요
- 문제 정의
- 대상 사용자
- 요구사항 요약
- MVP 범위
- 주요 기능 설명
- 기술 스택 선정 이유
- 시스템 구조
- 구현 과정
- 오류와 해결 과정
- 테스트 결과
- 한계점
- 향후 개선 사항
- 느낀 점 또는 학습 내용

보고서 작성 원칙:

- 단순히 기능 목록만 나열하지 않습니다.
- 왜 이런 기능을 만들었는지 설명합니다.
- 수업 시간 안에서 어떤 범위를 MVP로 정했는지 밝힙니다.
- 오류가 있었다면 원인과 해결 과정을 간단히 기록합니다.
- 테스트 결과는 실행 여부와 기대 결과 중심으로 작성합니다.
- 과장하지 않고 실제 구현된 내용을 기준으로 작성합니다.

## 5. 기술 스택 정리 방식

기술 스택은 사용한 기술과 사용 이유를 함께 정리합니다.

권장 형식:

| 구분 | 기술 | 사용 이유 |
| --- | --- | --- |
| Backend | FastAPI | Python 기반으로 API를 빠르게 만들고 Swagger 테스트를 지원합니다. |
| AI API | OpenAI API 또는 Gemini API | 텍스트 생성, 요약, 분류 등 AI 기능 구현에 사용합니다. |
| Agent Workflow | LangChain 또는 LangGraph | 여러 단계의 AI 처리 흐름을 구성할 때 확장할 수 있습니다. |
| Vector DB | FAISS 또는 ChromaDB | 문서 검색 또는 RAG 기능 확장에 사용할 수 있습니다. |
| Frontend | React, Next, Nuxt 중 선택 | 사용자 입력 화면과 결과 화면을 구현합니다. |
| Environment | python-dotenv | API Key 등 환경 변수를 안전하게 불러옵니다. |

주의:

- 실제로 사용하지 않은 기술을 사용한 것처럼 쓰지 않습니다.
- 확장 예정인 기술은 "향후 적용 가능" 또는 "확장 예정"으로 구분합니다.
- 기술 이름만 나열하지 말고 프로젝트에서 맡은 역할을 설명합니다.

## 6. 프로젝트 구조 설명 방식

프로젝트 구조는 폴더 트리와 설명 표를 함께 작성합니다.

예시:

```text
project/
├── backend/
│   ├── app/
│   │   ├── main.py
│   │   ├── api/
│   │   ├── services/
│   │   └── agents/
│   └── requirements.txt
├── frontend/
└── README.md
```

설명 표:

| 경로 | 설명 |
| --- | --- |
| `backend/app/main.py` | FastAPI 애플리케이션 진입점입니다. |
| `backend/app/api/` | API 라우터와 요청/응답 schema를 관리합니다. |
| `backend/app/services/` | AI API 호출과 핵심 처리 로직을 관리합니다. |
| `backend/app/agents/` | Agent Workflow 또는 LangGraph 확장 코드를 관리합니다. |
| `frontend/` | 사용자 화면을 관리합니다. |

작성 원칙:

- 모든 파일을 설명하지 않아도 됩니다.
- 핵심 폴더와 주요 파일 위주로 설명합니다.
- 수강생이나 평가자가 프로젝트 구조를 빠르게 이해할 수 있게 작성합니다.

## 7. 실행 방법 작성 기준

실행 방법은 실제로 따라 할 수 있는 순서로 작성합니다.

포함할 항목:

- 저장소 복제
- 가상환경 생성
- 패키지 설치
- 환경 변수 설정
- 서버 실행
- 프론트엔드 실행
- 접속 주소

백엔드 실행 예시:

```bash
cd backend
python -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
uvicorn app.main:app --reload
```

Windows PowerShell 예시:

```powershell
cd backend
python -m venv .venv
.venv\Scripts\Activate.ps1
pip install -r requirements.txt
uvicorn app.main:app --reload
```

환경 변수 안내 예시:

```bash
cp .env.example .env
```

주의:

- 실제 `.env` 값을 README에 쓰지 않습니다.
- OS별 명령어가 다르면 구분해서 작성합니다.
- 실행 확인 URL을 함께 적습니다.

## 8. API 사용 방법 작성 기준

API 사용 방법은 Swagger UI 기준과 예시 Request/Response를 함께 작성합니다.

포함할 항목:

- Swagger 접속 주소
- API Endpoint
- HTTP Method
- Request Body 예시
- Response Body 예시
- 오류 응답 예시

예시:

```text
Swagger UI: http://127.0.0.1:8000/docs
```

API 명세 예시:

| 항목 | 내용 |
| --- | --- |
| Method | `POST` |
| Endpoint | `/api/summarize` |
| 설명 | 입력 텍스트를 요약합니다. |

Request 예시:

```json
{
  "text": "오늘 회의에서는 고객 피드백 분석 일정을 논의했습니다."
}
```

Response 예시:

```json
{
  "summary": "고객 피드백 분석 일정이 논의되었습니다.",
  "message": "요약이 완료되었습니다."
}
```

주의:

- 예시에는 민감 정보를 넣지 않습니다.
- 실제 응답과 다른 내용을 쓰지 않습니다.
- 오류 케이스도 최소 1개 이상 작성하면 좋습니다.

## 9. 테스트 결과 작성 기준

테스트 결과는 기능이 실제로 동작했는지 확인한 기록입니다.

포함할 항목:

- 테스트 일자
- 테스트 환경
- 테스트 항목
- 입력값
- 기대 결과
- 실제 결과
- 통과 여부
- 비고

권장 형식:

| 테스트 항목 | 입력 | 기대 결과 | 실제 결과 | 통과 여부 |
| --- | --- | --- | --- | --- |
| 정상 입력 요약 | 회의록 텍스트 | 요약 결과 반환 | 요약 결과 반환 | 통과 |
| 빈 입력 처리 | 빈 문자열 | 오류 메시지 반환 | 오류 메시지 반환 | 통과 |

작성 원칙:

- 테스트하지 않은 기능을 통과했다고 쓰지 않습니다.
- 실패한 테스트가 있다면 원인 또는 후속 작업을 적습니다.
- 자동 테스트가 없다면 수동 확인 결과를 작성합니다.
- API는 Swagger UI 또는 curl 테스트 결과를 기록할 수 있습니다.

## 10. 향후 개선 사항 작성 기준

향후 개선 사항은 MVP에서 제외한 기능과 실제 사용을 위해 필요한 개선점을 정리합니다.

작성 기준:

- MVP에서 제외한 기능을 우선 적습니다.
- 실제 사용성을 높이는 개선점을 적습니다.
- 기술적 개선과 기능적 개선을 구분합니다.
- 너무 막연한 표현보다 구체적인 개선 방향을 작성합니다.

좋은 예시:

```text
- 파일 업로드 기능을 추가해 회의록 텍스트 파일을 직접 업로드할 수 있도록 개선합니다.
- 요약 결과 저장 기능을 추가해 이전 분석 결과를 다시 확인할 수 있도록 개선합니다.
- LangGraph를 적용해 입력 분석, 요약 생성, 결과 검증 단계를 분리합니다.
```

나쁜 예시:

```text
- 더 좋게 만들기
- AI 고도화
```
