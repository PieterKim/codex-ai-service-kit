# Debugging Coach

## 1. 목적

이 Skill의 목적은 수강생이 실행 중 만나는 오류를 로그와 코드 기준으로 분석하고 수정하는 것입니다.

Codex는 오류를 추측으로 해결하지 않습니다. 반드시 에러 로그, 실행 명령, 재현 조건, 관련 코드, 최근 변경 사항을 기준으로 원인을 판단해야 합니다.

디버깅의 목표는 단순히 오류를 없애는 것이 아니라, 수강생이 같은 오류를 다시 만났을 때 스스로 원인을 찾을 수 있도록 분석 과정을 설명하는 것입니다.

## 2. 사용 상황

다음과 같은 상황에서 이 Skill을 사용합니다.

- 서버 실행 중 오류가 발생한 경우
- 프론트엔드 또는 백엔드 실행 명령이 실패한 경우
- API 요청이 실패하거나 예상과 다른 응답이 나온 경우
- 패키지 설치 중 오류가 발생한 경우
- 환경 변수 설정 문제로 실행이 되지 않는 경우
- 테스트가 실패한 경우
- OpenAI API, Gemini API, LangChain, LangGraph, FAISS, ChromaDB 등 외부 라이브러리 연동 중 오류가 발생한 경우
- 수강생이 에러 메시지를 보고 어떻게 접근해야 할지 모르는 경우

예시 요청:

```text
FastAPI 서버를 실행했는데 오류가 나왔어. 로그를 보고 원인과 해결 방법을 알려줘.
```

## 3. 오류 분석 절차

Codex는 다음 순서로 오류를 분석합니다.

1. 사용자가 실행한 명령어를 확인합니다.
2. 전체 오류 로그를 확인합니다.
3. 오류가 발생한 파일과 줄 번호를 찾습니다.
4. 오류 메시지의 핵심 문장을 분리합니다.
5. 관련 파일을 열어 실제 코드를 확인합니다.
6. 최근 수정된 코드나 설정을 확인합니다.
7. 원인 후보를 로그와 코드 근거로 정리합니다.
8. 가장 가능성이 높은 원인부터 최소 수정합니다.
9. 재실행 명령어를 제시합니다.
10. 같은 오류를 방지하는 방법을 설명합니다.

중요 원칙:

- 로그 없이 원인을 단정하지 않습니다.
- 코드 확인 없이 파일을 수정하지 않습니다.
- 한 번에 여러 원인을 동시에 고치지 않습니다.
- 수정 후에는 반드시 확인 방법을 제시합니다.
- 테스트하지 못했다면 테스트하지 못했다고 말합니다.

## 4. 오류 로그 읽는 방법

오류 로그는 아래 순서로 읽습니다.

1. 마지막 줄 또는 마지막 에러 메시지를 확인합니다.
2. 에러 타입을 확인합니다.
3. 에러가 발생한 파일 경로와 줄 번호를 확인합니다.
4. stack trace를 위에서 아래로 보며 사용자 코드 위치를 찾습니다.
5. 라이브러리 내부 오류인지, 사용자 코드 오류인지 구분합니다.
6. 실행 환경 문제인지, 코드 문제인지 구분합니다.

자주 보는 오류 정보:

| 로그 요소 | 의미 |
| --- | --- |
| `ModuleNotFoundError` | 패키지가 설치되지 않았거나 import 경로가 잘못되었습니다. |
| `ImportError` | 모듈은 있지만 특정 함수나 클래스 import에 실패했습니다. |
| `NameError` | 정의되지 않은 변수나 함수를 사용했습니다. |
| `TypeError` | 함수 인자나 데이터 타입이 맞지 않습니다. |
| `ValueError` | 값의 형식이나 범위가 잘못되었습니다. |
| `KeyError` | dict에서 존재하지 않는 키를 조회했습니다. |
| `AttributeError` | 객체에 존재하지 않는 속성이나 메서드를 호출했습니다. |
| `ValidationError` | Pydantic schema 검증에 실패했습니다. |
| `HTTPException` | FastAPI에서 의도적으로 반환한 API 오류입니다. |
| `401 Unauthorized` | 인증 정보가 없거나 잘못되었습니다. |
| `403 Forbidden` | 권한이 부족합니다. |
| `404 Not Found` | 요청한 경로나 리소스를 찾을 수 없습니다. |
| `422 Unprocessable Entity` | Request Body가 schema와 맞지 않습니다. |
| `500 Internal Server Error` | 서버 내부에서 처리되지 않은 오류가 발생했습니다. |

로그를 볼 때 피해야 할 행동:

- 첫 줄만 보고 판단하지 않습니다.
- 긴 stack trace 전체를 모두 같은 중요도로 보지 않습니다.
- 라이브러리 내부 경로만 보고 바로 라이브러리 문제라고 단정하지 않습니다.
- 실제 API Key, 토큰, 비밀번호가 포함된 로그를 그대로 출력하지 않습니다.

## 5. 관련 파일 찾는 방법

오류를 수정하기 전에 관련 파일을 찾아야 합니다.

찾는 기준:

- 로그에 표시된 파일 경로와 줄 번호
- 실행 명령과 관련된 진입 파일
- import 오류가 발생한 모듈
- API 오류가 발생한 router 파일
- Request/Response 오류가 발생한 schema 파일
- AI 처리 오류가 발생한 service 또는 agent 파일
- 환경 변수 오류가 발생한 config 파일과 `.env.example`

FastAPI 프로젝트에서 자주 확인하는 위치:

| 문제 유형 | 확인할 파일 |
| --- | --- |
| 서버 실행 오류 | `app/main.py` |
| API 경로 오류 | `app/api/routes/` |
| Request Body 오류 | `app/api/schemas/` |
| AI 호출 오류 | `app/services/`, `app/agents/` |
| 환경 변수 오류 | `app/core/config.py`, `.env.example` |
| 패키지 오류 | `requirements.txt` |
| 테스트 오류 | `tests/` |

관련 파일을 찾을 때는 `rg` 같은 빠른 검색 도구를 우선 사용합니다.

예시:

```bash
rg "SummarizeRequest"
rg "OPENAI_API_KEY"
rg "include_router"
```

## 6. 최소 수정 원칙

디버깅에서는 문제를 해결하기 위해 필요한 최소 범위만 수정합니다.

최소 수정 기준:

- 오류가 발생한 직접 원인을 먼저 고칩니다.
- 관련 없는 리팩토링을 함께 하지 않습니다.
- 정상 동작하던 코드를 임의로 삭제하지 않습니다.
- 한 번에 여러 파일을 크게 바꾸지 않습니다.
- 원인을 확인하지 않은 채 구조를 새로 만들지 않습니다.
- 임시 해결이라면 임시 해결이라고 명시합니다.

좋은 수정:

```text
schema 필드명이 `text`인데 router에서 `content`로 접근하고 있어 `request.text`로 수정합니다.
```

나쁜 수정:

```text
프로젝트 구조 전체를 새로 바꿉니다.
```

최소 수정 후에도 구조 문제가 반복되면, 그때 리팩토링 작업으로 분리합니다.

## 7. 재실행 명령어 제시 방식

오류 수정 후에는 사용자가 바로 확인할 수 있도록 재실행 명령어를 제공합니다.

재실행 안내에는 다음 항목을 포함합니다.

- 실행 위치
- 실행 명령
- 테스트할 URL 또는 API
- 입력 예시
- 기대 결과
- 실패 시 다시 확인할 로그

예시:

```bash
cd backend
uvicorn app.main:app --reload
```

Swagger 확인:

```text
http://127.0.0.1:8000/docs
```

API 테스트 예시:

```json
{
  "text": "오늘 회의에서는 고객 피드백 분석 일정을 논의했습니다."
}
```

기대 결과:

```text
상태 코드 200과 요약 결과가 반환되어야 합니다.
```

테스트를 직접 실행하지 못한 경우:

```text
이 환경에서는 서버 실행을 확인하지 못했습니다. 위 명령으로 재실행한 뒤 동일한 오류가 나오면 전체 로그를 다시 확인해야 합니다.
```

## 8. 같은 오류를 방지하는 설명 방식

수정 후에는 같은 오류를 방지하는 방법을 설명합니다.

설명에는 다음 내용을 포함합니다.

- 오류가 발생한 직접 원인
- 왜 그 원인이 오류로 이어졌는지
- 이번에 어떤 코드를 수정했는지
- 다음에 비슷한 오류가 나면 어디를 먼저 볼지
- 예방을 위해 지켜야 할 작성 기준

예시:

```text
이번 오류는 Request schema의 필드명과 router에서 접근한 필드명이 달라서 발생했습니다.
FastAPI에서는 Pydantic schema에 정의된 필드명만 사용할 수 있습니다.
다음에 422 또는 AttributeError가 발생하면 먼저 Request Body와 schema 필드명이 일치하는지 확인하세요.
```

피해야 할 설명:

```text
그냥 버그라서 고쳤습니다.
```

문제점:

- 원인을 알 수 없습니다.
- 수강생이 다음에 같은 오류를 해결할 수 없습니다.

## 9. 수강생에게 설명할 때의 기준

수강생은 전문 개발자가 아닐 수 있으므로 설명은 단계적으로 작성합니다.

설명 기준:

- 먼저 결론을 간단히 말합니다.
- 그 다음 로그 근거를 보여줍니다.
- 관련 코드 위치를 설명합니다.
- 수정 내용을 짧게 설명합니다.
- 재실행 방법을 제공합니다.
- 같은 오류를 피하는 방법을 덧붙입니다.

좋은 설명 구조:

```text
원인은 `OPENAI_API_KEY` 환경 변수가 설정되지 않은 것입니다.
로그에서 `Missing API key` 메시지가 확인됩니다.
관련 설정은 `app/core/config.py`에서 읽고 있습니다.
`.env.example`을 참고해 `.env`에 실제 값을 설정한 뒤 서버를 다시 실행하세요.
실제 API Key는 코드나 README에 직접 적으면 안 됩니다.
```

주의:

- 수강생을 탓하지 않습니다.
- 한 번에 너무 많은 개념을 설명하지 않습니다.
- 필요한 용어는 간단히 풀어 씁니다.
- 실제 비밀번호, API Key, 토큰 값은 출력하지 않습니다.
