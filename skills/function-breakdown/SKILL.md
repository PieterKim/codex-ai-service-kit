# Function Breakdown

## 1. 목적

이 Skill의 목적은 요구사항 정의서를 실제 개발 가능한 기능 단위로 분해하는 것입니다.

Codex는 사용자의 요구사항을 바탕으로 큰 기능을 작은 세부 기능으로 나누고, 각 기능의 입력, 처리, 출력, 구현 우선순위, 관련 Agent/API를 정리해야 합니다.

이 Skill의 결과물은 이후 MVP 설계, 프로젝트 구조 생성, API 구현, AI Agent Workflow 설계의 기준 자료로 사용됩니다.

## 2. 사용 상황

다음과 같은 상황에서 이 Skill을 사용합니다.

- 요구사항 정의서가 작성된 후 기능 목록을 정리해야 할 때
- 사용자가 "무엇부터 구현해야 할지 모르겠다"고 말할 때
- 서비스 아이디어가 너무 커서 작은 개발 작업으로 나눠야 할 때
- API 기능과 AI Agent 기능을 구분해야 할 때
- MVP에 포함할 기능 후보를 정리해야 할 때
- 팀원별 작업 분담을 위해 기능 단위가 필요할 때

예시 요청:

```text
요구사항 정의서를 바탕으로 실제 구현할 기능을 작게 나눠줘.
```

## 3. 작업 절차

Codex는 다음 순서로 기능을 분해합니다.

1. 요구사항 정의서에서 프로젝트 목표와 핵심 기능을 확인합니다.
2. 사용자의 입력 데이터와 출력 결과를 확인합니다.
3. 전체 기능을 화면, API, AI 처리, 데이터 저장, 문서화 등 대분류로 나눕니다.
4. 각 대분류를 구현 가능한 세부 기능으로 나눕니다.
5. 각 세부 기능에 대해 입력, 처리, 출력을 정리합니다.
6. 각 기능이 일반 API 기능인지 AI Agent 기능인지 구분합니다.
7. MVP에 포함할 기능과 후순위 기능을 구분합니다.
8. 구현 우선순위를 높음, 중간, 낮음으로 표시합니다.
9. 기능 분해표를 Markdown 표로 작성합니다.
10. 마지막에 구현 순서 제안을 간단히 덧붙입니다.

작업 중 요구사항이 부족하면 Codex는 임의로 확정하지 않고 "가정" 또는 "확인 필요"로 표시합니다.

## 4. 기능 분해 기준

기능은 너무 크지도, 너무 작지도 않게 나누어야 합니다.

좋은 기능 단위의 기준:

- 하나의 목적이 명확합니다.
- 입력과 출력이 구분됩니다.
- 구현 완료 여부를 확인할 수 있습니다.
- 테스트하거나 수동 검증할 수 있습니다.
- 한 번의 작업 요청으로 구현 가능한 크기입니다.

너무 큰 기능 예시:

```text
AI 분석 시스템 구현
```

더 좋은 분해 예시:

```text
사용자 텍스트 입력 받기
입력 텍스트 길이 검증하기
AI 요약 요청 보내기
요약 결과 화면에 표시하기
오류 메시지 보여주기
```

기능을 나눌 때는 다음 관점으로 확인합니다.

- 사용자가 직접 보는 화면 기능인가?
- 서버에서 처리하는 API 기능인가?
- AI 모델이 판단, 생성, 요약, 분류하는 기능인가?
- 데이터 저장 또는 조회가 필요한 기능인가?
- 테스트와 검증이 필요한 기능인가?
- README나 보고서에 설명해야 하는 기능인가?

## 5. API 기능과 AI Agent 기능 구분 방법

API 기능과 AI Agent 기능은 역할이 다릅니다.

### API 기능

API 기능은 클라이언트와 서버 사이에서 요청을 받고, 데이터를 검증하고, 결과를 반환하는 기능입니다.

API 기능으로 분류하는 경우:

- 사용자의 요청을 받습니다.
- 입력값을 검증합니다.
- 파일 업로드를 처리합니다.
- 데이터베이스에 저장하거나 조회합니다.
- AI 처리 결과를 클라이언트에 반환합니다.
- 인증, 상태 확인, 결과 조회 같은 일반 서버 기능입니다.

예시:

```text
POST /summaries 요청 받기
업로드 파일 검증하기
요약 결과 저장하기
분석 결과 목록 조회하기
```

### AI Agent 기능

AI Agent 기능은 AI가 여러 단계의 판단, 도구 사용, 반복 처리, 상태 관리 등을 수행하는 기능입니다.

AI Agent 기능으로 분류하는 경우:

- 입력 내용을 분석해 다음 행동을 결정합니다.
- 검색, 요약, 분류, 생성 등 여러 AI 작업을 순서대로 수행합니다.
- 외부 도구나 API를 선택적으로 호출합니다.
- 사용자의 목표에 맞춰 결과를 조합합니다.
- 실패했을 때 대체 흐름이 필요합니다.

예시:

```text
사용자 질문 의도 분류하기
필요한 자료 검색 여부 판단하기
검색 결과를 바탕으로 답변 생성하기
회의록에서 결정 사항과 할 일을 분리 추출하기
```

구분 기준:

| 질문 | API 기능 | AI Agent 기능 |
| --- | --- | --- |
| 단순 요청/응답인가? | 예 | 아니오 |
| 판단이나 생성이 필요한가? | 보통 아니오 | 예 |
| 여러 단계의 AI 처리가 필요한가? | 아니오 | 예 |
| 외부 도구 선택이 필요한가? | 보통 아니오 | 예 |
| 결과 전달이 주된 역할인가? | 예 | 아니오 |

## 6. MVP 기능과 후순위 기능 구분 방법

MVP는 서비스의 핵심 가치를 보여주는 최소 기능입니다.

MVP에 포함할 기능 기준:

- 서비스의 핵심 문제 해결에 직접 필요합니다.
- 사용자가 입력을 넣고 결과를 확인할 수 있습니다.
- 수업 기간 안에 구현 가능합니다.
- 테스트 가능한 단위입니다.
- 없으면 서비스가 성립하지 않습니다.

후순위 기능 기준:

- 있으면 좋지만 없어도 핵심 기능은 동작합니다.
- 구현 시간이 많이 걸립니다.
- UI 편의성 개선에 가깝습니다.
- 관리자 기능, 로그인, 결제, 알림, 고급 통계처럼 부가 기능입니다.
- 배포 자동화나 운영 편의 기능입니다.

예시:

| 기능 | 분류 | 이유 |
| --- | --- | --- |
| 텍스트 입력 후 AI 요약 결과 보기 | MVP | 서비스 핵심 가치입니다. |
| 요약 결과를 PDF로 다운로드 | 후순위 | 결과 확인은 가능하므로 필수는 아닙니다. |
| 사용자 로그인 | 후순위 | 개인화가 핵심이 아니라면 첫 버전에 필요하지 않습니다. |
| 오류 메시지 표시 | MVP | 사용자가 실패 원인을 알아야 합니다. |

## 7. 산출물 표 형식

Codex는 기능 분해 결과를 아래 표 형식으로 작성합니다.

```markdown
| 대분류 | 세부 기능 | 입력 | 처리 | 출력 | 구현 우선순위 | 관련 Agent/API |
| --- | --- | --- | --- | --- | --- | --- |
|  |  |  |  |  |  |  |
```

각 열의 작성 기준은 다음과 같습니다.

| 열 | 작성 기준 |
| --- | --- |
| 대분류 | 화면, API, AI 처리, 데이터, 테스트, 문서화 등 큰 묶음을 작성합니다. |
| 세부 기능 | 실제 구현 가능한 작은 기능 이름을 작성합니다. |
| 입력 | 사용자가 제공하거나 시스템이 받는 데이터를 작성합니다. |
| 처리 | 기능 내부에서 수행할 작업을 작성합니다. |
| 출력 | 사용자 또는 다른 기능에 전달되는 결과를 작성합니다. |
| 구현 우선순위 | 높음, 중간, 낮음 중 하나로 작성합니다. |
| 관련 Agent/API | API, Agent, Frontend, Database, Documentation 등 관련 영역을 작성합니다. |

## 8. 예시

아래는 "AI 회의록 요약 서비스"의 기능 분해 예시입니다.

| 대분류 | 세부 기능 | 입력 | 처리 | 출력 | 구현 우선순위 | 관련 Agent/API |
| --- | --- | --- | --- | --- | --- | --- |
| 화면 | 회의록 텍스트 입력 | 회의록 텍스트 | 입력 폼에서 텍스트를 받음 | 입력된 텍스트 | 높음 | Frontend |
| API | 요약 요청 API | 회의록 텍스트 | 입력값 검증 후 AI 처리 요청 | 요약 결과 응답 | 높음 | API |
| AI 처리 | 핵심 내용 요약 | 회의록 텍스트 | 주요 안건과 결정 사항 추출 | 요약 문장 | 높음 | Agent |
| AI 처리 | 할 일 목록 추출 | 회의록 텍스트 | 담당자, 할 일, 기한 후보 추출 | 할 일 목록 | 높음 | Agent |
| 화면 | 결과 표시 | 요약 결과 | 요약, 결정 사항, 할 일 목록을 구분해 표시 | 결과 화면 | 높음 | Frontend |
| 오류 처리 | 빈 입력값 처리 | 빈 텍스트 | 입력값 검증 및 안내 메시지 생성 | 오류 메시지 | 높음 | API |
| 데이터 | 요약 결과 저장 | 요약 결과 | 결과를 로컬 또는 DB에 저장 | 저장된 결과 | 중간 | Database/API |
| 문서화 | README 작성 | 구현 기능 목록 | 실행 방법과 사용 예시 정리 | README.md | 중간 | Documentation |
| 부가 기능 | PDF 다운로드 | 요약 결과 | 결과를 PDF 파일로 변환 | PDF 파일 | 낮음 | API |
| 부가 기능 | 사용자 로그인 | 사용자 계정 정보 | 계정 인증 및 사용자별 결과 관리 | 로그인 상태 | 낮음 | API/Database |

구현 순서 제안:

1. 회의록 텍스트 입력
2. 요약 요청 API
3. 핵심 내용 요약
4. 할 일 목록 추출
5. 결과 표시
6. 오류 처리
7. README 작성
8. 저장, PDF 다운로드, 로그인 등 후순위 기능
