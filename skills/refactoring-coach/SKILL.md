# Refactoring Coach

## 1. 목적

이 Skill의 목적은 기존 기능을 유지하면서 코드 구조를 개선하는 것입니다.

Codex는 동작 중인 기능을 깨뜨리지 않는 범위에서 중복 코드, 지나치게 긴 함수, 역할이 섞인 파일, 부족한 타입 힌트, 불명확한 예외 처리를 개선해야 합니다.

수강생 프로젝트에서는 과도한 구조 변경보다 이해 가능한 수준의 리팩토링을 우선합니다. 리팩토링 후에는 반드시 기존 기능이 그대로 동작하는지 확인할 수 있는 실행 방법 또는 테스트 방법을 제시해야 합니다.

## 2. 사용 상황

다음과 같은 상황에서 이 Skill을 사용합니다.

- 기능은 동작하지만 코드가 너무 길거나 복잡한 경우
- 같은 코드가 여러 파일이나 함수에 반복되는 경우
- router, service, schema의 역할이 섞여 있는 경우
- 함수 이름이나 변수 이름이 의미를 알기 어려운 경우
- 타입 힌트가 없어 데이터 흐름을 이해하기 어려운 경우
- 예외 처리가 부족해 오류 원인을 알기 어려운 경우
- README나 보고서에 설명하기 어려울 정도로 구조가 정리되지 않은 경우
- 테스트나 수동 확인 전에 코드 정리가 필요한 경우

예시 요청:

```text
기능은 동작하는데 코드가 지저분해. 이해하기 쉽게 리팩토링해줘.
```

## 3. 리팩토링 원칙

Codex는 다음 원칙을 따릅니다.

- 기존 기능을 유지합니다.
- 변경 범위를 작게 유지합니다.
- 한 번에 많은 구조를 바꾸지 않습니다.
- 사용자가 이해할 수 있는 수준으로 개선합니다.
- 기존 프로젝트의 스타일과 구조를 우선 따릅니다.
- 불필요한 추상화나 디자인 패턴을 추가하지 않습니다.
- 리팩토링 전 관련 파일을 먼저 확인합니다.
- 리팩토링 후 실행 방법과 테스트 방법을 제공합니다.

좋은 리팩토링:

```text
router에 있던 AI 호출 로직을 service 함수로 분리하고, router는 요청과 응답만 담당하게 만듭니다.
```

피해야 할 리팩토링:

```text
작은 MVP 프로젝트에 복잡한 계층 구조, 인터페이스, 팩토리 패턴을 한 번에 추가합니다.
```

## 4. 기존 기능 유지 기준

리팩토링은 기능 변경이 아닙니다.

기존 기능 유지 기준:

- 기존 API 경로가 유지됩니다.
- 기존 Request/Response 형식이 유지됩니다.
- 기존 실행 명령이 가능하면 유지됩니다.
- 사용자가 보던 결과가 동일하게 유지됩니다.
- 오류 메시지의 의미가 유지됩니다.
- 테스트 또는 수동 확인 결과가 리팩토링 전과 동일합니다.

기능 변경이 필요한 경우에는 리팩토링과 분리해서 사용자에게 설명합니다.

예시:

```text
현재 작업은 리팩토링입니다.
API 응답 형식을 바꾸는 것은 프론트엔드에 영향을 줄 수 있으므로 별도 기능 변경으로 다루는 것이 좋습니다.
```

## 5. 중복 코드 제거 기준

중복 코드는 같은 의도와 같은 동작이 반복될 때 제거합니다.

제거 대상:

- 같은 입력 검증 로직이 여러 router에 반복됨
- 같은 AI API 호출 코드가 여러 함수에 반복됨
- 같은 오류 응답 생성 코드가 반복됨
- 같은 환경 변수 읽기 코드가 여러 파일에 흩어져 있음
- 같은 데이터 변환 로직이 반복됨

중복 제거 방식:

- 공통 함수로 분리합니다.
- service 계층으로 이동합니다.
- config 파일에서 한 번만 관리합니다.
- schema를 재사용합니다.
- 너무 작은 중복은 무리하게 추상화하지 않습니다.

주의:

- 우연히 비슷해 보이지만 의미가 다른 코드는 합치지 않습니다.
- 수강생이 이해하기 어려운 과도한 공통화는 피합니다.
- 공통 함수 이름은 역할이 명확해야 합니다.

## 6. 파일 역할 분리 기준

파일은 역할에 따라 분리합니다.

FastAPI 프로젝트 기준:

| 파일 또는 폴더 | 역할 |
| --- | --- |
| `app/main.py` | FastAPI 앱 생성, router 등록 |
| `app/api/routes/` | API endpoint 정의 |
| `app/api/schemas/` | Request/Response Pydantic schema 정의 |
| `app/services/` | 핵심 처리 로직, AI API 호출, 외부 API 연동 |
| `app/agents/` | Agent Workflow 또는 LangGraph 흐름 |
| `app/core/config.py` | 환경 변수와 설정값 관리 |
| `app/utils/` | 여러 곳에서 쓰는 작은 공통 함수 |
| `tests/` | 테스트 코드 |

분리 기준:

- router에는 HTTP 요청과 응답 처리만 둡니다.
- service에는 실제 기능 로직을 둡니다.
- schema에는 데이터 구조와 검증 규칙을 둡니다.
- config에는 환경 변수 처리를 둡니다.
- utils에는 특정 도메인에 강하게 묶이지 않는 공통 함수를 둡니다.

피해야 할 구조:

```text
하나의 파일에 API 라우터, AI 호출, 환경 변수 처리, 데이터 변환, 예외 처리가 모두 들어 있음
```

## 7. 타입 힌트 추가 기준

타입 힌트는 코드의 입력과 출력을 이해하기 쉽게 만듭니다.

추가 기준:

- 함수의 매개변수와 반환값에 타입을 표시합니다.
- list, dict 같은 컬렉션은 가능한 내부 타입까지 표시합니다.
- Optional 값은 `str | None`처럼 명확히 표시합니다.
- Pydantic schema와 service 함수 사이의 데이터 타입을 맞춥니다.
- 타입 힌트가 너무 복잡해지면 구조를 단순화합니다.

예시:

```python
def summarize_text(text: str) -> dict[str, object]:
    ...
```

더 명확한 예시:

```python
def summarize_text(text: str) -> SummarizeResponse:
    ...
```

주의:

- 타입 힌트는 실행 결과를 바꾸기 위한 것이 아닙니다.
- 수강생이 이해하기 어려운 복잡한 generic 타입은 피합니다.
- 타입 힌트를 추가한 뒤 import가 필요한지 확인합니다.

## 8. 예외 처리 추가 기준

예외 처리는 오류가 났을 때 원인을 이해하고 안전하게 응답하기 위해 추가합니다.

추가 기준:

- 사용자가 잘못 입력할 수 있는 경우
- 외부 AI API 호출이 실패할 수 있는 경우
- 환경 변수가 없을 수 있는 경우
- 파일 업로드 또는 파일 읽기가 실패할 수 있는 경우
- 벡터 DB나 외부 저장소 연결이 실패할 수 있는 경우

FastAPI 예외 처리 기준:

| 상황 | 권장 상태 코드 |
| --- | --- |
| 입력값이 비어 있음 | `400 Bad Request` |
| Request Body 형식이 잘못됨 | `422 Unprocessable Entity` |
| 리소스를 찾을 수 없음 | `404 Not Found` |
| 외부 AI API 호출 실패 | `502 Bad Gateway` |
| 서버 내부 오류 | `500 Internal Server Error` |

주의:

- 모든 예외를 무조건 `except Exception`으로 숨기지 않습니다.
- 실제 API Key, 토큰, 비밀번호를 오류 메시지에 포함하지 않습니다.
- 사용자에게 보여줄 메시지와 개발자가 볼 로그를 구분합니다.
- 예외 처리를 추가하면서 원래 정상 흐름을 바꾸지 않습니다.

## 9. 테스트 방법

리팩토링 후에는 기존 기능이 유지되는지 확인해야 합니다.

테스트 방법:

- 기존 테스트가 있으면 먼저 실행합니다.
- 테스트가 없다면 수동 확인 절차를 작성합니다.
- API 기능은 Swagger UI 또는 curl로 확인합니다.
- 프론트엔드 기능은 브라우저에서 주요 흐름을 확인합니다.
- 오류 처리 변경이 있다면 실패 케이스도 확인합니다.

FastAPI 확인 예시:

```bash
cd backend
uvicorn app.main:app --reload
```

Swagger 확인:

```text
http://127.0.0.1:8000/docs
```

수동 확인 항목:

- 서버가 정상 실행되는가?
- 기존 API 경로가 그대로 동작하는가?
- 정상 입력에서 기존과 같은 결과가 나오는가?
- 잘못된 입력에서 적절한 오류 메시지가 나오는가?
- README의 실행 방법이 여전히 맞는가?

테스트를 실행하지 못했다면 그 이유를 명확히 설명합니다.
